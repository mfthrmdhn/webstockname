# Plan 04-01: Backend API Routes & Schema for Product Edit/Delete

**Objective:** Implement PATCH and DELETE endpoints for product management with full audit trail. Add is_active field to Product model for soft-delete capability. Extend logAction utility to capture PRODUCT_UPDATE (delta only) and PRODUCT_DELETE (full snapshot) actions.

**Purpose:** Enable superadmin to edit all product fields safely and delete only products with no sales history. All changes are immutably logged with before/after values for accountability and compliance.

**Output:** 
- PATCH /api/products/{id} endpoint with Zod validation, atomicity, and audit logging
- DELETE /api/products/{id} endpoint with sales history check and audit logging
- is_active field on Product model (for Phase 4 soft-delete flow and future archival)
- Extended logAction utility with new action types

### Task 3: Create PATCH /api/products/[id] endpoint with delta audit logging

<read_first>
- app/api/products/route.ts (existing POST endpoint pattern: auth, rbac, zod validation, prisma query, logAction)
- middleware/auth.ts (authMiddleware pattern)
- middleware/rbac.ts (rbacMiddleware pattern)
- lib/audit/logger.ts (logAction usage)
- prisma/schema.prisma (Product model fields to validate)
</read_first>

<action>
Create app/api/products/[id]/route.ts with PATCH handler:

1. Apply authMiddleware and rbacMiddleware(['SUPERADMIN']) before any logic (same pattern as POST /api/products)

2. Define updateProductSchema with Zod:
```typescript
const updateProductSchema = z.object({
  name: z.string().min(1).max(255).optional(),
  sku: z.string().min(3).max(50).optional(),
  sellingPrice: z.number().positive().optional(),
  cost: z.number().min(0).optional(),
  storeQty: z.number().min(0).optional(),
  warehouseQty: z.number().min(0).optional(),
  category: z.string().max(100).optional().nullable(),
})
```
All fields optional — superadmin can update any subset of fields.

3. Implement PATCH logic:
   - Extract product ID from route params: request.nextUrl.pathname.split('/').pop()
   - Validate request body with updateProductSchema
   - Query current product: prisma.product.findUnique({where: {id}}) — if not found, return 404
   - Validate SKU uniqueness only if SKU is being changed: if (data.sku && data.sku !== currentProduct.sku) check for duplicate
   - Build before/after metadata objects containing ONLY changed fields:
     ```typescript
     const changed: Record<string, unknown> = {}
     const before: Record<string, unknown> = {}
     const after: Record<string, unknown> = {}
     
     for (const [key, value] of Object.entries(updateData)) {
       if (currentProduct[key as keyof Product] !== value) {
         changed[key] = true
         before[key] = currentProduct[key as keyof Product]
         after[key] = value
       }
     }
     
     if (Object.keys(changed).length === 0) {
       return NextResponse.json({message: 'No changes'}, {status: 200})
     }
     ```
   - Update product in transaction:
     ```typescript
     const updated = await prisma.$transaction(async (tx) => {
       return await tx.product.update({
         where: {id},
         data: updateData,
       })
     })
     ```
   - Log audit entry: `logAction(userId, 'PRODUCT_UPDATE', 'PRODUCT', id, {before, after, changedFields: Object.keys(changed)})`
   - Return updated product with 200

4. Handle errors:
   - Invalid ID format → 400
   - Product not found → 404 with message
   - SKU already exists → 400 "SKU already in use"
   - Database errors → 500

Return response includes id, name, sku, sellingPrice, cost, storeQty, warehouseQty, createdAt (no before/after in response, only in audit log).
</action>

### Task 4: Create DELETE /api/products/[id] endpoint with sales history check and soft-delete fallback

<read_first>
- app/api/products/[id]/route.ts (PATCH handler from Task 3, for auth/rbac pattern)
- prisma/schema.prisma (Product model, Sale model relationship)
- lib/audit/logger.ts (logAction usage for PRODUCT_DELETE)
</read_first>

<action>
Add DELETE handler to app/api/products/[id]/route.ts (same file as PATCH):

1. Apply authMiddleware and rbacMiddleware(['SUPERADMIN']) (same pattern as PATCH)

2. Extract product ID from route params

3. Check for sales history:
   ```typescript
   const salesCount = await prisma.saleItem.count({
     where: {productId: id}
   })
   ```
   (Count saleItems referencing this product; if > 0, product has sales history)

4. If salesCount > 0:
   - Return 400 with error response:
     ```json
     {
       "error": "Cannot delete product",
       "reason": "This product has sales history",
       "salesCount": X,
       "alternative": "Use soft-delete instead (set is_active to false)"
     }
     ```
   - Do NOT perform deletion; this blocks hard-delete if sales exist
   - Do NOT log deletion attempt to audit_log (logging happens only after successful deletion)

5. If salesCount === 0:
   - Fetch full product snapshot before deletion (for audit log):
     ```typescript
     const product = await prisma.product.findUnique({where: {id}})
     const snapshot = {
       id: product.id,
       name: product.name,
       sku: product.sku,
       sellingPrice: product.sellingPrice,
       cost: product.cost,
       storeQty: product.storeQty,
       warehouseQty: product.warehouseQty,
       createdAt: product.createdAt,
     }
     ```
   - Delete product in transaction:
     ```typescript
     await prisma.$transaction(async (tx) => {
       await tx.product.delete({where: {id}})
     })
     ```
   - Log deletion: `logAction(userId, 'PRODUCT_DELETE', 'PRODUCT', id, {before: snapshot, after: null})`
   - Return 200 with success message:
     ```json
     {
       "message": "Product deleted successfully",
       "id": "...",
       "name": "..."
     }
     ```

6. Handle errors:
   - Invalid ID format → 400
   - Product not found → 404
   - Database errors → 500

TypeScript note: Prisma.product.delete() requires product to exist or throws error; catch and return 404.
</action>
