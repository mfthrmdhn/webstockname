  <action>
Replace the fallback JWT secret initialization with an explicit check that throws if the env var is not set.

**Current code (lines 4-6):**
```typescript
const JWT_SECRET = new TextEncoder().encode(
  process.env.JWT_SECRET || 'dev-secret-change-in-production'
)
```

**Fixed code:**
```typescript
const rawSecret = process.env.JWT_SECRET
if (!rawSecret) {
  throw new Error('JWT_SECRET environment variable must be set')
}
const JWT_SECRET = new TextEncoder().encode(rawSecret)
```

**Why this approach:**
- Explicit check makes the requirement obvious: `if (!rawSecret)` is unmistakable
- Error message is specific and actionable
- Throwing at module load time (not at request time) means the error surfaces immediately at startup
- Next.js middleware running at the edge will show 500 error during initialization, making misconfiguration visible in logs
- No fallback to an insecure default: fail closed, not open
- Production deployments that omit JWT_SECRET will be immediately obvious when the application refuses to start

**Timing:**
- This code runs when middleware.ts is imported (at application startup)
- If JWT_SECRET is missing, the error prevents the module from loading
- Next.js edge runtime will report the error in deployment logs, preventing silent failure
- Local development without JWT_SECRET will also fail loudly, ensuring developers don't accidentally commit with missing env var

**Key difference from current code:**
- Current: `process.env.JWT_SECRET || 'dev-secret...'` — silently accepts missing env var and uses public fallback
- Fixed: `if (!rawSecret) throw new Error(...)` — fails explicitly and loudly
