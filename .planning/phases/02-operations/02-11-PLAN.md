  <action>
Replace the fallback JWT secret initialization with an explicit check that throws if the env var is not set. This ensures the application fails to start if the secret is missing, rather than silently using a public fallback.

**Current state (BROKEN):**
```typescript
const JWT_SECRET = new TextEncoder().encode(
  process.env.JWT_SECRET || 'dev-secret-change-in-production'
)
```

The `||` operator provides a fallback that is:
1. Hardcoded in the source code
2. Publicly visible in version control
3. Same across all deployments
4. Known to be unsafe ("change-in-production")

If JWT_SECRET is missing in production, the app uses this known secret, allowing attackers to forge valid JWT tokens.

**Fixed state:**
```typescript
const rawSecret = process.env.JWT_SECRET
if (!rawSecret) {
  throw new Error('JWT_SECRET environment variable must be set')
}
const JWT_SECRET = new TextEncoder().encode(rawSecret)
```

**Why this approach:**
- Explicit check makes the requirement unmistakable: `if (!rawSecret)` is self-documenting
- Error message is specific and actionable: developers/operators know exactly what's missing
- Throwing at module load time (not at request time) means the error surfaces immediately at startup
- Next.js middleware module loading happens before any requests, so errors appear in init logs
- No fallback to an insecure default: fail closed (explicit error), not open (silent fallback)
- Production deployments that omit JWT_SECRET will be immediately obvious when the application refuses to start
- Local development without JWT_SECRET will also fail loudly, ensuring developers don't accidentally deploy with missing secrets

**Timing:**
- This code runs when middleware.ts is imported (application startup)
- If JWT_SECRET is missing, the error prevents the module from loading
- Next.js deployment logs will show the error immediately
- Application will not start, preventing any requests from being processed
- Zero window where unauthed requests could be accepted

**Key difference from current code:**
- Current: `process.env.JWT_SECRET || 'dev-secret...'` — silently accepts missing env var and uses publicly-known fallback
- Fixed: `if (!rawSecret) throw new Error(...)` — fails explicitly and loudly, prevents startup
